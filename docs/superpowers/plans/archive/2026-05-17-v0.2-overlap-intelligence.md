# v0.2 Overlap Intelligence Follow-Up

## Summary

Phase 2 now closes the v0.1 explainability gap: recommended skills receive `activation_overlap` and `activation_conflict` warnings when existing registry overlap/conflict data matters to `recommend` and `explain`.

The next implementation slice is v0.2 overlap intelligence: richer, deterministic explanations for when overlapping skills are complementary, redundant, or conflicting. This should remain advisory, local-first, and inspectable.

## Key Changes

- Extend overlap analysis beyond shared triggers/tags into deterministic bundle evidence such as policy similarity, shared capabilities, common parents, and provider provenance.
- Keep `findOverlaps()` and `findConflicts()` as the core public analysis entry points, but enrich their returned details only after defining the contract.
- Preserve current CLI/MCP envelope shapes. Any recommendation-specific ambiguity should remain in `recommendation.warnings`.
- Avoid embeddings, semantic routing, hidden activation, workflow execution, or daemon behavior.

## Test Plan

- Core tests for deterministic overlap classification and conflict ordering.
- CLI tests for `clew-cli overlaps`, `clew-cli conflicts`, `clew-cli recommend`, and `clew-cli explain`.
- MCP tests proving enriched advisory warnings stay on recommendations and top-level registry/request warnings remain unchanged.
- Contract tests only if the public overlap/conflict return shape changes.

## Assumptions

- v0.1 warning codes remain open strings; no schema enum change is required for new advisory warning codes.
- Recommendation signal types stay limited to activation evidence (`trigger`, `tag`, `agents_md`, `repo_signal`) unless a separate schema decision expands them.
- SQLite overlap/conflict tables remain rebuildable derived state from filesystem skill bundles.
